# /run — 전체 프로젝트 자동화 파이프라인

사용자가 `/run`을 입력하면 아래 전체 흐름을 자동으로 실행한다.
팀장 에이전트가 브리프를 분석하고, 필요한 하위 에이전트를 순서대로 배치해 완성 결과물까지 만든다.
**디자인 포함 시 7단계 파이프라인(레퍼런스 퍼스트 + 팀장 컨펌 게이트)을 반드시 따른다.**

---

## 실행 순서

### Step 1. 브리프 수집
사용자에게 아래를 물어본다 (한 번에):
```
1. 프로젝트명 또는 브랜드명
2. 만들고 싶은 것 (예: 브랜드 전략, 앱, 디자인, 또는 전부)
3. 타깃 고객
4. 희망 레퍼런스 URL (있으면 — 없으면 생략 가능)
5. 마감 기한 또는 우선순위
6. 시작 단계 선택 (선택사항):
   A. 전체 (전략→디자인→개발→QA→배포) [기본값]
   B. 디자인부터 (레퍼런스→에셋→Figma→개발→QA→배포)
   C. 개발부터 (개발→QA→배포)
   D. QA+배포만
   E. 배포만
7. 참고할 이전 프로젝트 (선택사항):
   - 이전 프로젝트명을 입력하면 해당 프로젝트의 전략/디자인/코드 패턴을 기반으로 진행
   - 예: "풀케어 DID 광고" → 해당 프로젝트의 컬러·폰트·레이아웃을 참조
```
→ 6번 미입력 시 전체(A) 자동 선택. 7번 미입력 시 새로 시작. 입력받은 즉시 분석 시작. 추가 질문 최소화.

**이전 프로젝트 참조 규칙:**
- 7번에 프로젝트명 입력 시 → 해당 프로젝트 폴더의 기획서·에셋 가이드·코드를 자동 탐지
- 디자인 파이프라인에서 참조 시 → 레퍼런스 수집 단축 (기존 에셋 가이드의 컬러·폰트를 시드로 활용)
- 개발 파이프라인에서 참조 시 → 기술 스택·Firebase 연동 패턴 자동 재사용
- 참조는 "기반"일 뿐 — 새 프로젝트의 고유 요구사항이 우선

**단계 스킵 규칙:**
- 선택한 시작 단계 이전의 에이전트는 호출하지 않음
- 이전 단계 산출물이 이미 있으면(기존 기획서, 에셋 등) 자동 감지하여 활용
- 이전 단계 산출물이 없으면 해당 단계 없이 진행 (에셋 가이드 없이 개발 등)

### Step 2. 작업 계획 수립 (자동)
브리프를 분석해 아래 형식으로 작업 계획을 즉시 출력:
```
📋 [프로젝트명] 작업 계획

1단계: strategy-planning — [작업 내용] → 산출물: 기획 브리프
2단계: design-reference-archiver — 시드 분석 + 유사 레퍼런스 확장 수집 → 산출물: 포인트 보고서
  ↳ ★ 팀장 컨펌 ① — 보고서 승인/수정 ★
3단계: design-asset-generator — 메인폰트 선정 + 에셋 생성 → 산출물: 에셋 적용 가이드
  ↳ ★ 팀장 컨펌 ② — 적용 가이드 승인/보완 ★
4단계: design-figma — 가이드대로 HTML 조립 + Figma 캡처 → 산출물: Figma 파일
  ↳ ★ 팀장 최종 검수 ★
5단계: program-development — [작업 내용] → 산출물: 실행 가능한 HTML

예상 산출물: X종
저장 위치: 01_baroezip/[프로젝트명]/ (바로이집) 또는 클로드/[프로젝트명]/ (기타)
```
→ 사용자에게 한 번 확인 후 실행 시작.

### Step 3. 에이전트 순서대로 자동 실행

**판단 기준 (필요한 단계만 포함):**
- 시장 조사/데이터 필요 → `research-crawler` 에이전트 (전략·기획의 팩트 근거 수집)
- 브랜드/전략 포함 → `strategy-planning` 에이전트 (research-crawler 데이터 활용)
- 디자인 포함 → 7단계 파이프라인 적용 (아래 참조)
- 앱/도구 포함 → `program-development` 에이전트 (전략+디자인 결과물 참고)
- 품질 검증 필요 → `qa-tester` 에이전트 (기능·반응형·보안·디자인 일치 검증)
- 배포 필요 → `deploy-manager` 에이전트 (Netlify/Firebase 자동 배포)
- SNS 업로드 포함 → `auto-login-executor` 에이전트 (완성물 업로드)

**디자인 파이프라인 (포함 시 필수):**
```
① strategy-planning (기획이 필요한 경우)
   ↓ 기획 브리프 + 시드 레퍼런스 URL
② design-reference-archiver (시드 분석 + 유사 레퍼런스 확장 수집)
   ↓ 디자인/구성 포인트 보고서
③ ★ 팀장 컨펌 ① ★ — 포인트 보고서 확인 → 승인 또는 수정 지시
   ↓ 컨펌된 디자인 포인트
④ design-asset-generator (메인폰트 선정 + 에셋 생성 + 에셋 적용 가이드)
   ↓ 에셋 적용 가이드 (폰트·컬러·에셋 목록·레이아웃)
⑤ ★ 팀장 컨펌 ② ★ — 적용 가이드 확인 → 승인 또는 보완 지시
   ↓ 컨펌된 에셋 + 가이드
⑥ design-figma (가이드대로 HTML 조립 → Figma 캡처 → 품질 확인)
   ↓ Figma 파일
⑦ ★ 팀장 최종 검수 ★ — 완료 또는 재캡처 지시
```

**에이전트 간 결과물 연계:**
- 전략기획 완료 → 기획 브리프를 레퍼런스 에이전트에 전달
- 시드 레퍼런스 URL → 레퍼런스 에이전트에 시드로 전달
- 레퍼런스 포인트 보고서 → 팀장 컨펌 후 에셋 에이전트에 전달
- 에셋 적용 가이드 (폰트·컬러·에셋·레이아웃) → 팀장 컨펌 후 Figma 에이전트에 전달
- 전략 + 디자인 결과물 → 개발 에이전트 컨텍스트에 자동 전달

### Step 4. 최종 보고
모든 에이전트 작업 완료 후 출력:
```
✅ [프로젝트명] 완료 보고

완성된 산출물:
📁 01_baroezip/[프로젝트명]/ (바로이집) 또는 클로드/[프로젝트명]/ (기타)
  ├── 전략기획/       [파일 목록]
  ├── 레퍼런스/       [포인트 보고서, 링크 모음]
  ├── 디자인/
  │   ├── assets/     [에셋 파일 목록]
  │   ├── 적용가이드.md
  │   └── *.html      [Figma 캡처용]
  └── 개발/           [실행 파일]

적용 정보:
  메인폰트: [폰트명]
  컬러 팔레트: 주색 #XXX | 보조색 #XXX | 배경 #XXX
  Figma 파일: [링크]

다음 액션 제안:
- [제안 1]
- [제안 2]
```

---

## 실행 원칙
- 각 단계 시작 전 "▶ [에이전트명] 시작합니다" 보고
- 각 단계 완료 후 "✅ [에이전트명] 완료 — [산출물 요약]" 보고
- **컨펌 게이트에서는 반드시 사용자에게 보고 후 승인/수정 확인**
- 중간에 막히면 멈추지 말고 가능한 부분까지 진행 후 보고
- 컨펌 게이트 외에는 단계 중간에 추가 확인하지 않음
