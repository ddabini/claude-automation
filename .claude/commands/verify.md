# /verify — 작업 결과 실제 반영 검증 + 자동 수정

**목적: "됐습니다"라는 보고가 거짓이 되지 않도록, 모든 변경사항을 직접 확인한다.**

에이전트가 완료를 보고해도 실제로는 미반영된 경우가 빈번하다.
이 스킬은 **보고 내용과 실제 상태를 대조**하고, 불일치 시 **직접 해결**한다.

---

## 실행 방식

### 자동 실행 (기본)
- leader.rules의 모든 작업 완료 시 **팀장 코멘트 작성 전** 자동 실행
- 사용자가 별도로 `/verify` 입력할 필요 없음

### 수동 실행
- 사용자가 `/verify` 입력 시 가장 최근 작업에 대해 즉시 검증 실행
- `/verify [파일경로]` — 특정 파일 지정 검증

---

## 검증 3단계

### Step 1. 파일 검증 — "수정한 내용이 실제로 파일에 존재하는가?"

**반드시 수행:**
1. 수정/생성했다고 보고한 **모든 파일**을 `Read` 도구로 다시 읽는다
2. 보고한 변경사항이 파일 내용에 **실제로 존재하는지** 확인한다
3. 검증 항목:
   - [ ] 파일이 존재하는가?
   - [ ] 보고한 코드/텍스트가 파일에 포함되어 있는가?
   - [ ] 문법 오류 없이 올바른 형태인가? (닫히지 않은 태그, 괄호 불일치 등)
   - [ ] 기존 코드를 실수로 덮어쓰거나 삭제하지 않았는가?

**검증 방법:**
```
보고: "pikbox.html에 YouTube 비교 기능 추가"
→ Read로 pikbox.html 열기
→ 해당 함수/변수/UI 요소가 실제 존재하는지 검색
→ 함수가 호출되는 곳까지 확인 (선언만 하고 연결 안 한 경우 방지)
```

### Step 2. 기능 검증 — "코드가 실제로 동작하는가?"

**웹 파일(HTML/JS)인 경우:**
1. Playwright를 사용하여 브라우저에서 실제 실행
2. 보고한 기능을 **직접 조작**하여 동작 확인:
   - 버튼 클릭 → 예상 결과 발생하는가?
   - 데이터 입력 → 저장/표시되는가?
   - UI 변경 → 화면에 반영되는가?
3. 콘솔 에러가 없는가?
4. 스크린샷을 찍어 시각적으로 확인

**비웹 파일(스크립트, 설정 등)인 경우:**
- 실행 가능하면 실행하여 결과 확인
- 설정 파일이면 문법 검증 (JSON parse, YAML lint 등)

**스킵 조건:**
- 단순 텍스트 수정 (README, 주석 등) → Step 1만으로 충분
- Playwright 연결 불가 → Step 1 결과만으로 판단 (사용자에게 Playwright 미사용 명시)

### Step 3. 배포 검증 — "라이브에 반영되었는가?"

**배포를 수행한 경우에만 실행:**
1. 배포 URL에 **캐시 버스팅** 파라미터 추가하여 접속
   ```
   curl -H 'Cache-Control: no-cache' "https://url.web.app?v=$(date +%s)"
   ```
2. 변경사항이 라이브에 반영되었는지 확인:
   - HTML 내용에 변경 키워드가 포함되는가?
   - Playwright로 접속하여 기능 동작 확인
3. **미반영 시**: Firebase CDN 캐시 문제 → 재배포 또는 캐시 무효화

## 불일치 발견 시 — 자동 수정 루프

### 핵심 원칙: 고칠 수 있으면 알아서 고쳐라

**사용자에게 실패를 보고하기 전에, 스스로 해결할 수 있는지 먼저 판단한다.**

| 상황 | 행동 |
|------|------|
| 코드 오류, 함수 미연결, 문법 실수, CSS 깨짐 등 | **즉시 직접 수정** → 재검증 → 통과 시 완료 보고 (수정 과정 간략 언급) |
| 배포 미반영, 캐시 문제 | **재배포 실행** → 재검증 |
| 에이전트가 빠뜨린 기능 | **직접 구현 또는 에이전트 재호출** → 재검증 |
| 파일 경로 오류, 에셋 누락 | **직접 수정/생성** → 재검증 |
| **새로운 허가/결제/가입이 필요** | ❌ **이때만 사용자에게 보고** |
| **사용자 판단이 필요한 방향성 결정** | ❌ **이때만 사용자에게 보고** |
| **외부 서비스 장애 (API 다운 등)** | ❌ **이때만 사용자에게 보고** |

```
검증 실패 감지
  ↓
내가 고칠 수 있는가? ─── YES ──→ 즉시 수정 (에이전트 재호출 포함)
  │                                  ↓
  │                              재검증 (Step 1~3)
  │                                  ↓
  │                            ┌─ 통과 → ✅ 완료 보고 (수정 내역 간략 포함)
  │                            └─ 실패 → 다른 접근으로 재시도 (최대 5회)
  │
  └── NO (허가/결제/가입/외부장애 필요) ──→ ❌ 사용자에게 실패 분석 보고
```

**자동 수정 범위 (사용자 허가 없이 즉시 실행):**
- 코드 수정 (버그 픽스, 누락 기능 추가, 리팩토링)
- 에이전트 재호출 (program-development, design-figma 등)
- 파일 생성/수정/삭제 (프로젝트 범위 내)
- 재배포 (기존 배포 설정 그대로)
- CSS/HTML/JS 수정
- 에셋 재생성

**사용자 보고가 필요한 경우 (= 내 권한 밖):**
- 새로운 API 키 발급, 유료 서비스 결제
- 새로운 계정 가입/로그인
- 사용자가 직접 외부 도구에서 조작해야 하는 작업
- 프로젝트 방향이 바뀌어야 하는 수준의 근본적 문제

**절대 규칙: 거짓 보고는 어떤 상황에서도 절대 허용하지 않는다.**
- 안 된 걸 "됐습니다"라고 하는 것은 사용자의 시간을 낭비시키는 최악의 행위다
- 확인하지 않은 것을 "확인했습니다"라고 하지 않는다
- 부분 성공이면 "부분 성공"이라고 정확히 말한다
- 실패하면 실패라고 말하고, **왜 실패했는지 분석**한다

## 검증 결과 보고 형식

### 깨끗하게 통과 시 (수정 없음)
```
✅ 검증 완료 — 파일 ✅ | 기능 ✅ | 배포 ✅(또는 해당없음)
```
→ 팀장 코멘트에 한 줄로 포함. 별도 보고 불필요.

### 자동 수정 후 통과 시 (스스로 고침)
```
✅ 검증 완료 (자동 수정 [N]건)
  - [무엇이 안 됐고 → 어떻게 고쳤는지] (간략히)
```
→ 사용자에게는 **고친 결과만 보여준다**. 중간 실패 과정을 길게 나열하지 않는다.
→ 사용자 입장: "알아서 잘 해결해서 완성된 상태로 가져왔구나" — 이것이 목표.

### 사용자 액션 필요 시 (내 권한 밖) — 실패 분석 보고서

**이 보고는 허가/결제/가입/외부장애 등 내가 해결할 수 없을 때만 발생한다.**
보고 시 반드시 3가지를 포함: ① 왜 실패 ② 뭐가 문제 ③ 어떻게 해결 가능

```
⏸️ 사용자 액션 필요
━━━━━━━━━━━━━━━━━━

■ 현재 상태: [완료된 부분] + [막힌 부분]

■ 막힌 이유: [왜 내가 해결할 수 없는지 — 쉬운 말로]
  - [구체적 원인: API 키 미발급 / 유료 서비스 / 계정 필요 / 외부 장애 등]

■ 사용자가 해주실 일:
  1. [구체적 단계 — 어디서 무엇을 하면 되는지]
  2. [필요한 정보를 알려주시면 이후 자동 진행]

■ 이후 계획: [사용자 액션 완료 후 내가 자동으로 할 일]
```

## 검증 범위 판단 기준

| 작업 유형 | Step 1 (파일) | Step 2 (기능) | Step 3 (배포) |
|-----------|:---:|:---:|:---:|
| 코드 수정 (HTML/JS/CSS) | ✅ 필수 | ✅ 필수 | 배포 시만 |
| 단순 텍스트 수정 (MD, 주석) | ✅ 필수 | — | — |
| 설정 파일 수정 | ✅ 필수 | ✅ 문법검증 | — |
| 디자인 (HTML 생성) | ✅ 필수 | ✅ 렌더링 확인 | — |
| 배포 작업 | ✅ 필수 | ✅ 필수 | ✅ 필수 |
| 기획서/전략서 | ✅ 필수 | — | — |

---

## 성능 최적화

- 수정 파일이 1~2개이고 단순 수정이면 → Step 1만 빠르게 실행 (30초 이내)
- 대규모 수정(5개+ 파일, 새 기능)이면 → 전체 3단계 실행
- Playwright 실행이 불필요하게 무거우면 → Read + Grep으로 코드 레벨 검증만 수행
- **검증 자체가 작업보다 오래 걸리지 않도록** 적절히 범위 조절

## 팀장 연동

이 스킬의 결과는 팀장 코멘트의 `✅ 품질` 항목에 자동 반영된다:
```
✅ 품질 — /verify 통과 (파일 ✅ | 기능 ✅ | 배포 ✅)
```
또는
```
✅ 품질 — /verify 수정 후 통과 (1차 실패: 함수 미연결 → 직접 수정 후 재검증 통과)
```
또는
```
❌ 품질 — /verify 실패 (3회 시도 후 미해결 — 사용자 확인 필요)
```
