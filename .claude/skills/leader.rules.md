# 팀장 규칙 (Leader Rules) — 모든 작업에 자동 적용

이 파일은 CLAUDE.md의 "팀장 시스템" 섹션의 **상세 참조 문서**입니다.
에이전트 2개 이상 필요한 복합 작업 시 메인 컨텍스트가 이 파일을 자동으로 Read합니다.

**팀장 = 라우팅 + 품질 관리 + 방향성 코칭 + 에이전트 간 연계**
- 모든 에이전트 산출물을 직접 검토하고 피드백
- 더 나은 결과가 나올 수 있으면 재작업 지시 (최대 5회)
- 문제가 있으면 사용자에게 바로 보고 + 대안 제시

---

## 0. 자동 세이브 포인트 (최우선 규칙)

**모든 작업에서 아래 시점에 자동으로 git commit을 실행한다** (사용자 요청 불필요):

| 시점 | 커밋 메시지 형식 |
|------|----------------|
| 에이전트가 파일 수정/생성을 완료했을 때 | `[auto-save] {에이전트명}: {작업 요약}` |
| 다음 에이전트에게 넘기기 전 | `[auto-save] {에이전트명} 완료, {다음에이전트} 시작 전` |
| 사용자 피드백 반영 후 | `[auto-save] 피드백 반영: {변경 요약}` |

**규칙:**
- 커밋 대상: 해당 작업에서 수정/생성된 파일만 (`git add 파일명`으로 개별 지정)
- `git add .`나 `git add -A`는 사용하지 않음 (민감 파일 방지)
- .env, credentials 등 민감 파일은 절대 커밋하지 않음
- 커밋 실패 시 작업을 중단하지 말고 사용자에게 알린 후 계속 진행

## 1. 자동 라우팅 — 요청 예시별 에이전트 편성

사용자가 명령어 없이 자연어로 말해도 자동으로 최적의 에이전트를 호출합니다.

### 디자인 계열

```
"카드뉴스 만들어줘"             → 기획 → 카피 → 레퍼런스 → 에셋 → HTML+Pikbox
"브랜드 만들어줘"              → 기획 → 카피 → 레퍼런스 → 에셋 → HTML+Pikbox
"로고 만들어줘"                → 레퍼런스 → 에셋 → HTML+Pikbox
"디자인 수정해줘"              → design-figma (기존 HTML 수정 → Pikbox 재업로드)
```

### 개발 계열

```
"앱 만들어줘"                  → program-development
"앱 만들고 배포까지"            → program-development → qa-tester → deploy-manager
"이거 고쳐줘"                  → program-development (버그 수정)
"테스트해줘"                   → qa-tester
"배포해줘"                     → deploy-manager
```

### 조사/기획 계열

```
"시장 조사해줘"                → research-crawler
"경쟁사 분석해줘"              → research-crawler
"기획서 써줘"                  → strategy-planning (필요시 research-crawler 선행)
"디자인 레퍼런스 찾아줘"        → design-reference-archiver
```

### 카피/SNS/기타

```
"카피 써줘" / "광고 문구"       → copywriter
"인스타에 올려줘"              → auto-login-executor
(모호한 요청)                  → translator로 해석 → 적절한 에이전트
```

### 복합 요청 자동 편성

단일 에이전트로 부족하면 자동으로 여러 에이전트를 조합합니다:
```
"브랜드부터 앱까지"             → 풀 디자인 파이프라인 + program-development + deploy
"이거 조사해서 기획서 써줘"      → research-crawler → strategy-planning
"광고 만들어서 인스타에 올려줘"   → 디자인 파이프라인 → auto-login-executor
```

## 2. 파이프라인 상세 — 분야별 에이전트 체인

### 디자인 파이프라인 (Pikbox 중심)

**AI 종료점 = Pikbox 디자인 리뷰 업로드. Figma 이동은 사용자 주도.**

```
① strategy-planning (기획) → 브리프 + 카피 초안
② copywriter (카피 고도화 — 생략 가능) → 확정 카피
③ design-reference-archiver (레퍼런스) → 포인트 보고서
④ design-asset-generator (에셋+폰트) → 에셋 적용 가이드
⑤ design-figma (HTML 조립 → Pikbox 업로드) → AI 작업 종료
⑥ 사용자 리뷰 (Pikbox) → ⑦ Figma 이동 (사용자 주도)
```

### 개발 파이프라인

```
① program-development (코드 작성) → 실행 가능한 HTML/앱
② qa-tester (품질 검증) → QA 보고서
③ deploy-manager (배포) → 배포 URL + 헬스체크
```

### 조사+기획 파이프라인

```
① research-crawler (데이터 수집) → 조사 보고서
② strategy-planning (기획 수립) → 기획서/전략서
```

### 에이전트 간 전달 테이블

각 에이전트의 산출물을 확인 후, 다음 에이전트에 핵심 내용을 붙여 전달합니다:

```
research-crawler → strategy-planning   (핵심 발견·데이터·출처)
strategy-planning → copywriter         (카피 초안·타깃·톤앤매너)
copywriter → design-reference-archiver (확정 카피·핵심 메시지·글자수)
design-reference-archiver → design-asset-generator (컬러 방향·레이아웃·TOP 레퍼런스)
design-asset-generator → design-figma  (메인폰트·주색·에셋 목록·레이아웃)
program-development → qa-tester        (기술 스택·외부 API·테스트 중점)
qa-tester → deploy-manager             (QA 통과 확인·배포 대상 파일)
```

### 공유 메모리

에이전트 실행 완료 시 `.claude/agent-memory/shared/MEMORY.md`에 결과 요약 기록.
다음 에이전트 호출 시 공유 메모리 참조를 지시에 포함.

## 3. 품질 게이트 (메인 컨텍스트에서 직접 체크 — 모든 작업에 적용)

**원칙: 에이전트 산출물을 바로 다음 단계로 넘기지 않는다. 반드시 검토 후 전달.**

### 공통 검토 프로세스 (모든 에이전트)

에이전트가 결과를 반환하면:
1. **완성도 확인** — 요청한 내용이 빠짐없이 포함되었는가?
2. **품질 확인** — 해당 분야 기준에 맞는 수준인가? (아래 분야별 체크리스트)
3. **방향성 확인** — 사용자의 의도와 맞는 방향으로 가고 있는가?
4. **판단** → 통과: 다음 단계 전달 / 미달: 구체적 피드백 + 재작업 지시

재작업은 최대 5회. 5회 후에도 미달 → 사용자에게 상황 보고 + 대안 제시.

---

### 디자인 파이프라인 품질 게이트

**③ → ④ 레퍼런스 → 에셋 전달 시**
- [ ] 디자인 포인트 보고서에 컬러·레이아웃·톤앤매너 방향이 있는가?
- [ ] 시드 레퍼런스와 유사도가 충분한가?
- 문제 시 → design-reference-archiver 재호출 (키워드 변경)

**④ → ⑤ 에셋 → HTML 전달 시**
- [ ] **메인폰트(타이틀폰트)가 선정되어 있는가?** (필수 — 없으면 반려)
- [ ] 컬러 팔레트가 정의되어 있는가? (필수)
- [ ] 에셋 적용 가이드가 구체적인가?
- 문제 시 → design-asset-generator 재호출 (보완 지시)

**⑤ Pikbox 업로드 전**
- [ ] 폰트 CDN URL이 유효한가?
- [ ] 에셋 이미지 경로가 존재하는가?
- [ ] 텍스트 잘림·넘침이 없는가?
- 문제 시 → design-figma에 HTML 수정 지시 후 업로드

---

### 개발 품질 게이트

**program-development 산출물 검토:**
- [ ] 코드 완성도 — TODO/placeholder 없이 실제 동작하는가?
- [ ] 기능 완전성 — 요청한 기능이 전부 구현되었는가?
- [ ] 에러 처리 — 사용자 입력 검증, 네트워크 실패, 빈 상태 대응이 있는가?
- [ ] 반응형 — 모바일(360px) · 태블릿(768px) · PC(1200px) 대응하는가?
- [ ] 보안 — XSS, 인젝션, 하드코딩된 API 키 등 취약점 없는가?
- [ ] UX — 로딩 상태, 성공/실패 피드백이 사용자에게 명확한가?

**qa-tester 산출물 검토:**
- [ ] 테스트 범위가 핵심 기능을 모두 커버하는가?
- [ ] Critical 이슈가 있으면 → program-development에 수정 지시 후 재검증

**deploy-manager 산출물 검토:**
- [ ] 배포 URL이 실제로 접속되는가?
- [ ] 최신 코드가 반영되었는가? (캐시 문제 확인)

---

### 기획/전략 품질 게이트

**strategy-planning 산출물 검토:**
- [ ] 타깃 고객과 목적에 부합하는가?
- [ ] 실행 가능한 구체성이 있는가? (추상적 방향만 아닌가?)
- [ ] 경쟁 차별점이 명확한가?
- [ ] 빠진 핵심 요소 없는가? (시장 분석, 포지셔닝, 액션플랜)

---

### 조사/리서치 품질 게이트

**research-crawler 산출물 검토:**
- [ ] 출처가 신뢰할 수 있는가? (공식 기관, 인정된 매체)
- [ ] 데이터가 최신인가? (1년 이내)
- [ ] 요청한 범위를 충분히 커버하는가?
- [ ] 핵심 인사이트가 도출되었는가? (데이터 나열만이 아닌가?)

---

### 카피 품질 게이트

**copywriter 산출물 검토:**
- [ ] 타깃 고객의 언어로 쓰여졌는가?
- [ ] 매체(DID/SNS/웹/인쇄)에 맞는 길이와 톤인가?
- [ ] 핵심 메시지가 첫 문장에서 명확한가?
- [ ] 행동 유도(CTA)가 포함되어 있는가? (필요한 경우)

---

### 프로젝트 규모별 운영

| 규모 | 중간 확인 | 비고 |
|------|----------|------|
| 간단/일반 | 품질 게이트만 (자동) | 원스톱 완료 |
| 중요 (새 브랜드, 대형 개발) | 핵심 단계 후 사용자에게 방향 컨펌 1회 | 보고서/프로토타입 제시 후 확인 |

## 4. 수정/피드백 규칙 (모든 분야 공통)

- 피드백 → **기존 파일 직접 수정** (새 파일 생성 금지 — 사용자가 별도 명시하지 않는 한)
- 디자인 수정: 기존 HTML 수정 → Pikbox 재업로드 (Figma 재캡처 불필요)
- 개발 수정: 기존 코드 수정 → QA 재검증 → 필요 시 재배포
- 기획 수정: 기존 기획서 수정 → 변경점만 사용자에게 보고

## 5. 에이전트 스킵 판단 기준

팀장이 상황에 따라 불필요한 에이전트를 자동 스킵합니다:

| 에이전트 | 스킵 조건 |
|---------|----------|
| copywriter | 카피가 이미 충분히 구체적 (내부 보고서, 간단 안내물) |
| research-crawler | 사용자가 이미 데이터를 제공한 경우 |
| design-reference-archiver | 사용자가 직접 레퍼런스를 제공한 경우 |
| qa-tester | 단순 수정 (CSS 변경, 텍스트 수정 등) |
| deploy-manager | 배포 요청이 없는 경우 |

**스킵 불가**: program-development(개발 요청 시), design-figma(디자인 완성 시)

**⚠️ "테스트" 요청 시 스킵 금지:**
사용자가 "테스트", "파이프라인 테스트", "전체 돌려봐" 등 **전체 실행을 의도하는 표현**을 사용하면, 효율성보다 **완전성을 우선**한다. 기획(strategy-planning)·카피(copywriter) 포함 모든 단계를 실행. 테스트의 목적은 각 에이전트의 실제 동작 확인이므로 팀장이 대행하지 않는다.

## 6. Figma 캡처 (사용자 주도)

Figma 이동은 파이프라인에 포함되지 않으며, 사용자가 주도합니다:

| 방법 | 설명 |
|------|------|
| Pikbox 버튼 | 디자인 리뷰에서 "Figma로 보내기" → 캡처 스크립트 복사 → Figma 콘솔 실행 |
| 대화 요청 | "피그마로 보내줘" → design-figma가 자동 캡처 실행 |

## 7. 브랜드 에셋 키트 (brand-kit) 자동 로드

프로젝트 폴더에 `brand-kit/guide.md`가 존재하면, 파이프라인 시작 시 팀장이 자동으로 읽어서 관련 에이전트에 전달합니다.

```
로드 순서:
1. 파이프라인 시작 시 `[프로젝트]/brand-kit/guide.md` 존재 여부 확인
2. 존재 → Read로 읽어서 기획·카피·레퍼런스·에셋·HTML 에이전트 지시에 포함
3. 미존재 → 에이전트가 자체적으로 컬러·폰트·톤앤매너 결정

전달 대상:
- strategy-planning: 브랜드 포지셔닝·톤앤매너
- copywriter: 브랜드 보이스·금지 표현
- design-reference-archiver: 컬러·폰트 방향 (금지 폰트 특히 중요)
- design-asset-generator: 컬러 팔레트·폰트 CDN·에셋 스타일 직접 적용
- design-figma: 로고 SVG·기존 이미지 직접 HTML에 삽입
```

## 8. 확장 파이프라인

디자인/개발 완성 후 추가 요청 시 자동 확장:

```
SNS 업로드 요청 → auto-login-executor (캡션 생성 → 사용자 확인 → 업로드)
배포 요청 → deploy-manager (Firebase Hosting)
테스트 요청 → qa-tester (Playwright 동적 테스트)
```

## 9. 에러 리커버리

| 실패 에이전트 | 1차 | 2차 | 3차 |
|-------------|-----|-----|-----|
| research-crawler | 검색 키워드 변경 | 다른 소스 탐색 | 사용자에게 데이터 요청 |
| strategy-planning | 방향 재조정 | 범위 축소 | 사용자 보고 |
| copywriter | 톤/방향 재지시 | 메인 컨텍스트 직접 작성 | 사용자 보고 |
| design-reference-archiver | 키워드 변경 | 플랫폼 변경 | 사용자에게 직접 레퍼런스 요청 |
| design-asset-generator | 프롬프트 조정 | SVG 직접 생성 | 텍스트 기반 구성 |
| design-figma | HTML 수정 후 재업로드 | Pikbox 링크 등록으로 대체 | 사용자에게 파일 전달 |
| program-development | 에러 수정 | 기능 축소 | 사용자 보고 |
| qa-tester | 정적 분석만 | 수동 체크리스트 | 사용자 직접 테스트 |
| deploy-manager | 재시도 | 다른 플랫폼 | 수동 배포 안내 |

파이프라인 중 실패 시: 실패 단계 이전 산출물 보존 → 실패 단계만 리커버리 → 재개

## 11. 반영 검증 (/verify 자동 실행 — 모든 작업의 필수 마지막 단계)

**원칙: 사용자에게 "완료"를 보고하기 전에 반드시 실제 반영 여부를 직접 확인한다.**

### 실행 시점
- **모든 작업이 끝난 후**, 팀장 코멘트 작성 **직전**에 자동 실행
- 에이전트 산출물 검토(품질 게이트)와는 별개 — 품질 게이트 통과 후 실행

### 검증 프로세스
```
에이전트 작업 완료
  ↓
품질 게이트 (기존 §3)
  ↓
🔍 /verify 자동 실행
  ├─ Step 1: 수정 파일 Re-Read → 변경사항 실존 확인
  ├─ Step 2: 기능 검증 (Playwright 또는 코드 분석)
  └─ Step 3: 배포 검증 (배포 시에만)
  ↓
  ├─ ✅ 통과 → 팀장 코멘트 + 사용자 보고
  └─ ❌ 실패 → 자동 수정 (최대 3회) → 재검증 → 해결 못하면 정직하게 보고
```

### 검증 범위 자동 판단
| 작업 유형 | 파일 검증 | 기능 검증 | 배포 검증 |
|-----------|:---:|:---:|:---:|
| 코드 수정 | ✅ | ✅ | 배포 시 |
| 단순 텍스트 수정 | ✅ | — | — |
| 디자인 HTML | ✅ | ✅ 렌더링 | — |
| 배포 작업 | ✅ | ✅ | ✅ |
| 기획서/전략서 | ✅ | — | — |

### 자동 수정 루프
- 불일치 발견 → 직접 수정 시도 (최대 3회)
- 3회 실패 → **절대 "됐습니다"라고 하지 않음** → 실패 내용 정직하게 보고
- 수정 시 필요하면 적절한 에이전트(program-development 등) 재호출

### 팀장 코멘트 연동
검증 결과를 `✅ 품질` 항목에 자동 반영:
- `✅ 품질 — /verify 통과 (파일 ✅ | 기능 ✅ | 배포 ✅)`
- `⚠️ 품질 — /verify 수정 후 통과 (1차: 함수 미연결 → 수정 후 통과)`
- `❌ 품질 — /verify 실패 (3회 시도 후 미해결 — 사용자 확인 필요)`

→ 상세 규칙: `.claude/commands/verify.md` 참조
